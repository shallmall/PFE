End-to-End Walkthrough: Heterogeneous Graph Transformer (HGTModel) Pipeline
This document is the master technical reference for the Heterogeneous Graph Transformer (HGTModel) fraud detection project. It traces every granular detail of the pipeline from the initial raw CSV to the final mathematical evaluation.

1. Data Ingestion & Preprocessing
(File: build_graph.py)

The system ingests public_reviews_dataset_cleaned.csv. Rather than parsing the natural language text of the reviews, the architecture relies exclusively on behavioral signals derived from just 4 raw features: review_rating, review_date, number_of_helpful, and number_of_photos.

Crucial Preprocessing Steps:
Chronological Sorting: The entire DataFrame is strictly sorted by review_date. This is critical. It prevents temporal leakage, ensuring that the model cannot "see into the future" during training.
Missing Data Imputation: Null numeric columns are aggressively filled (e.g., helpful votes default to 0, ratings default to a neutral 3.0).
Integer Mapping (Scaffolding): Neural networks cannot read string IDs like "A2M1P5X". The pipeline extracts all unique reviewer_id, asin (product), and review_id strings and maps them to consecutive integers (0, 1, 2...) based on chronological appearance. These integers are used purely to draw the graph edges and are never passed as features.
2. Feature Engineering & Graph Construction
(File: build_graph.py)

The system calculates 13 highly predictive features across the three node types.

TIP

Why Engineer Features? We calculate metrics like mean_rating_given upfront rather than letting the GNN learn it. This saves neural network capacity, prevents the signal from blurring inside the 64-dim hidden layer, and provides perfect global context even when analyzing local subgraphs.

Reviewer Nodes (7 Extracted Features)
review_volume: Total historical reviews.
mean_rating_given: Average star rating.
rating_variance: Variance in ratings (fraudsters often have near-zero variance, giving only 1s or 5s).
helpfulness_ratio: Average helpful votes per review.
media_propensity: Total photos uploaded.
max_daily_velocity: The absolute maximum number of reviews posted in a single calendar day (a massive red flag for bot activity).
avg_target_product_bimodal_index: Average bimodal index of targeted products (precomputed 4-hop structural shortcut).
📦 Product Nodes (3 Extracted Features)
total_review_weight: Total reviews received.
bimodal_index: The ratio of extreme ratings (1 and 5) to moderate ratings (2, 3, 4). Products targeted by fraud rings often exhibit a polarized "U-shaped" distribution.
arrival_velocity: Total reviews divided by the product's lifespan in days.
📝 Review Nodes (4 Mixed Features)
review_rating (Raw)
number_of_helpful (Raw)
number_of_photos (Raw)
rating_dev_from_mean (Extracted): The difference between this specific review's rating and the product's overall average.
All 13 features are individually normalized using StandardScaler to ensure stable neural network gradients.

Building the HeteroData Object
Edges: Using the Integer Maps, directed edges are drawn for (reviewer, authored, review) and (review, targeted, product).
Bidirectional Flow: T.ToUndirected() adds reverse edges, allowing information to flow backwards (e.g., from Product back to Reviewer).
The Scale: As shown by our visualization scripts, the final graph contains 719,465 Nodes and 763,468 Edges.
3. The Dual-Supervision Masking Strategy
(File: build_graph.py)

The graph is split temporally: 80% Train (Oldest), 10% Validation (Newer), 10% Test (Newest). However, the labeling strategy differs fundamentally between the two classification heads:

IMPORTANT

Semi-Supervised Reviewers vs. Fully-Supervised Products

Reviewers (PU-Learning): The Reviewer dataset is massively imbalanced and mostly unknown. Out of 334,342 total reviewers, a staggering 89.00% (297,578) are Unlabeled, 6.86% (22,922) are known Fake, and 4.14% (13,842) are known Honest. Because so many bots go undetected, we use Positive-Unlabeled (PU) Learning. Unlabeled reviewers are labeled -1 and mathematically masked out of the loss calculation. The model uses the entire graph of 334k reviewers to learn the 4-hop structure, but only uses the 11% known labeled reviewers to grade itself and calculate gradients.
Products: Unflagged products use .fillna(0), assuming innocence. Every single product in the training split calculates loss. Because fraud attacks on products are obvious (massive spikes in volume), unflagged products serve as a highly reliable baseline of "normalcy".
4. The Heterogeneous Graph Transformer (HGTModel) Architecture
(File: model.py)

The core network is the Heterogeneous Graph Transformer (HGTModel built with HGTConv).

Phase 1: Feature Projection: Three separate Linear layers take the disparate feature sizes (6, 3, 4) and project them all into a unified hidden_channels size of 64.
Phase 2: Message Passing: Two layers of HGTConv allow the network to see 2 hops away (e.g., Product -> Review -> Reviewer).
Attention Mechanism (4 Heads): The model dynamically scores every edge. It learns to assign a massive attention weight (e.g., 0.95) to a suspicious review from a new account, while dropping the attention (e.g., 0.05) for a normal review.
Interleaved with ReLU activations and Dropout(0.3).
Phase 3: Multi-Task Classification: The network splits. The 64-dim Product embedding and the 64-dim Reviewer embedding are routed to their own separate, single-neuron Linear layers to output a raw fraud probability logit.
5. Training, Loss & Optimization
(File: train.py)

Fraud datasets suffer from extreme class imbalance. In the labeled Reviewer subset, the distribution actually leans towards fake (62% Fake vs 38% Honest), whereas the Product subset is over 90% Honest. train.py combats this with two heavy-hitting mathematical techniques:

Dynamic Class Weights: The script calculates the exact ratio of Honest-to-Fake nodes in the current training split (torch.sum(y == 0) / torch.sum(y == 1)). This ratio is passed directly to the loss function as pos_weight. If there are 10x more honest products than fake products, the loss function multiplies the penalty for missing a fake product by 10x!
Binary Focal Loss: Instead of standard Cross-Entropy, Focal loss (alpha=0.25, gamma=2.0) dynamically adjusts its attention. It heavily penalizes the model when it gets hard, anomalous cases wrong, while mathematically ignoring the millions of easy, obvious honest nodes that would otherwise drown out the gradients.
The Causality Loop:

Event Horizon Pruning: To prevent static graph temporal edge leakage across splits, train.py dynamically masks out any review events (and associated bipartite edges) occurring after the chronological training horizon cutoff (80% of total reviews). This guarantees the GNN aggregates strictly historical signals during gradient calculation.
Early Stopping: Monitors Validation Loss and halts training if no improvement is seen for 20 epochs, saving the optimal weights to best_model.pth.
6. Threshold Tuning & Final Evaluation
(File: train.py)

Standard ML uses a generic 0.5 probability threshold. Because we care deeply about catching fraud, the pipeline sweeps thresholds from 0.01 to 0.99 against the Validation Set to find the exact decimal that maximizes the F1-Fake score.

In our converged causal run (min_delta = 1e-4, early stopping at epoch 164), the optimal probability thresholds tuned on Validation were:

Product Threshold: 0.53
Reviewer Threshold: 0.33
Using these optimal thresholds on the unseen Test Set (339 Products, 34 Reviewers):

Product Head: Accuracy: 0.8879 | ROC-AUC: 0.9305 | PR-AUC: 0.9834
Catch rate on fake products (Recall): 0.8652 (231/267) with 0.9914 Precision.
Reviewer Head: Symmetrical 0.7647 across Accuracy, Precision, Recall, and F1-Score for both honest and fake classes.
